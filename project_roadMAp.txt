    CREATE TABLE categories (
        id SERIAL PRIMARY KEY,
        name VARCHAR (100) UNIQUE NOT NULL
    );

    CREATE TABLE suppliers(
        id SERIAL PRIMARY KEY,
        name VARCHAR (150) NOT NULL,
        contact_email VARCHAR(150)
    );

    CREATE TABLE products(
        id SERIAL PRIMARY KEY,
        sku VARCHAR(150) UNIQUE NOT NULL,
        name VARCHAR(150) NOT NULL,
        category_id INT REFERENCES categories(id) ON DELETE SET NULL,
        supplier_id INT REFERENCES suppliers(id) ON DELETE SET NULL,
        quantity INT NOT NULL DEFAULT 0 CHECK (quantity >= 0),
        reorder_level INT NOT NULL DEFAULT 5,
        unit_price NUMERIC(10, 2) NOT NULL,
        created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
    );
    CREATE TABLE stock_movement (
        id SERIAL PRIMARY KEY,
        prodoct_id INT REFERENCES products(id) ON DELETE CASCADE,
        change_type VARCHAR (10) CHECK (change_type IN ('restock', 'sale', 'adjustment')),
        note VARCHAR (255),
        created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
    );

            const result = await pool.query('SELECT * FROM categories (or suppliers) ORDER BY name');
            res.json(result.rows);
        });

        router.post('/', async (req, res) => {
            const { name } = req.body;
            if (!name) return res.status(400).json({ error: 'Name required' })
            const result = await pool.query('INSERT INTO categories (or suppliers) (name) VALUES ($1) RETURNING *')[name];
            res.status(201).json(result.rows[0]);
        });

    B: now move to the product routes (products.routes.js)
    in our case we have 3 things to manage search, filter and lower stock so insted of make each one have it routes and API me make the 3 in 1 go in the products routes js
    here a full example : 
        import e from "express";
        import pool from "../db";

        const router = e.Router();

        // GET /api/products?search=usb&category_id=2&lowStockOnly=true

        router.get('/', async (req, res) => {
            const { search, category_id, lowStockOnly } = req.query;
            const conditions = [];
            const values = [];
            if (search) {
                values.push(`%${search}%`);
                conditions.push(`(p.name ILIKE $${values.length} or p.sku ILIKE $${values.length}`)
            }
            if (category_id) {
                values.push(category_id)
                conditions.push(`p.category_id = $${values.length}`)
            }
            if (lowStockOnly === 'true') {
                conditions.push('p.quantity <= p.reorder.level')
            }
            const where = conditions.length ? `WHERE ${conditions.join(' AND ')}` : "";
            const result = await pool.query(`
                SELECT p.*, c.name AS category_name, s.name AS supplier_name,
                (p.quantity <= p.reorder_level) AS low_stock
                FROM products p
                LEFT JOIN categories c ON p.category_id = c.id
                LEFT JOIN suppliers s ON p.supplier_id = s.id
                ${where}
                ORDER BY p.name`, values);
            res.json(result.rows);
        })
        const { search, category_id, lowStockOnly } = req.query
            → req.query holds everything after the ? in the URL, e.g. /products?search=usb&category_id=2. All three filters are optional — none of this code runs if the frontend sends none of them.
        conditions / values arrays built up one 'if' at a time
            → Each active filter appends both a SQL fragment (to conditions) and its actual value (to values), keeping them in sync by array position — values.length always matches the $N placeholder just added.
        ILIKE
            → A case-insensitive LIKE — so a search for 'usb' also matches 'USB Cable'. The % wildcards around the search term mean it can match anywhere in the text, not just the start.
        conditions.length ? `WHERE ${conditions.join(' AND ')}` : ''
            → If no filters were sent, this becomes an empty string and the query simply selects everything — the WHERE clause only appears when there's something to filter by.
        LEFT JOIN categories / LEFT JOIN suppliers
            → A LEFT JOIN keeps every product row even if it has no category or supplier (NULL foreign key) — an INNER JOIN would silently hide those products, which would be a serious bug in an inventory app.
        (p.quantity <= p.reorder_level) AS low_stock
            → A computed column — Postgres evaluates this boolean expression for every row and returns it as a field called low_stock, so the frontend doesn't need to recalculate the same comparison itself.
    C: moving now to restock and sale endpoints now the POST in product.routes.js :
        Both wrap the quantity update and the movement log in a single database transaction — if either write fails, both roll back.
        This route follows the exact same BEGIN / COMMIT / ROLLBACK shape as restock, with two important additions.
        SELECT quantity FROM products WHERE id = $1 FOR UPDATE
            → FOR UPDATE locks this specific row for the rest of the transaction. Any other transaction trying to sell the same product has to wait until this one finishes (COMMIT or ROLLBACK) before it can read or change that row.
        if (product.rows[0].quantity < quantity) { ROLLBACK; return 400 }
            → The core business rule of the whole project: you cannot sell more than you have. This check happens on the server, inside the transaction, using the freshly-locked value — not trusting anything the frontend might have displayed.
